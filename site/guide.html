Welcome to rederive and SIR. My name is Lane; I'm the creator. This is not a labor of love or a "fun" project. This, to me, is a necessity. In a world of Shai-Hulud worms, mile-a-minute AI slop, and developers who blindly trust LLM output, I needed a tool to fight the onslaught. rederive is that tool. SIR is the vehicle.

The origin is interesting. It came from a theoretical exercise that turned into something much more: semcom, a tool that goes from semantic input to LLVM IR — the low-level code a machine actually runs — and then straight to a binary. The high-level concept is that the only purpose of a programming language is to convert human intent into byte code. In a world of agentic coding, I theorized that the programming language is an unnecessary abstraction.

In that process, a very real problem showed up: I can't read LLVM IR. It's virtually impossible to intuit what the IR does. I'm certain people smarter than me can do it — but not over a large project. So the question became: how do I know that my LLVM IR is correct? The answer came from the world of ML. When a model is trained, it learns from one set of data, and then held-out tests — examples kept hidden from it — verify the output.

The insight that created SIR and rederive is that LLM-based coding should follow the same process. The difference is that correctness can be checked by passing the same set of unit tests. In LLVM, it's an input → output system. In a higher-level language, it's an expect — a check that says what the answer should be.

You can think of everything here as test-driven development on steroids. The fundamentals come from classic engineering. Sharp eyes will notice concepts cribbed from compiler design, Forth, Smalltalk, and machine learning. We stand on the shoulders of giants!

— Lane, creator of rederive

Now, one thing to carry into the examples. rederive is built for AI coding agents; most of the time your agent runs it and you never see it work. But now and then it stops and asks you a question. When it does, it is not asking what you would like — rederive never guesses. A question means it hit a real choice it cannot make for you, and it needs one thing:

What is the source of truth here? Every question rederive asks is that same question, wearing a different coat. Once you see that, the questions stop being confusing.

And one word you'll meet everywhere: everything rederive writes down about what a piece of code does — its exact inputs, outputs, and quirks — is called a SIR. The letters stand for “Semantic Intermediate Representation,” but the plain meaning is the one you already have: the contract — the exact list of what the code does, not how it's written. When you see “SIR” anywhere in rederive, this is it. (The Spec page is this same thing, zoomed all the way in.)

The rest of this guide shows you what all of that feels like, through real examples. Every number and every line of code below is from an actual run.

Chapter 1

The invoice generator

Want to run this yourself while you read? Paste the prompt below into your coding agent (Claude Code, Cursor, or similar). It kicks off the exact build this chapter walks through — then read on to see what each step means.

▶ Follow along — paste into your coding agent
Set up rederive, then build a small invoice generator with me,
leaf by leaf, as verified ZERO-dependency units.

If you're Claude Code, install the toolkit first:
  /plugin marketplace add rederive/sir-toolkit
  /plugin install sir@sir-toolkit
(Other agents: the docs are at rederive.ai and
github.com/rederive/sir-toolkit.)

The app takes line items (description, quantity, unit price),
a currency, and a tax rate, and returns a formatted invoice
total.

Break it into named jobs — round money, format currency,
format the date. For each job: find the packages people
normally reach for, run them all on the same inputs, and where
they DISAGREE on a real choice (how to round a half-cent, how
to format money for different currencies), STOP and ask me to
decide — never guess. Where they agree, that's the contract.

packages. When it's done, show me the verification (quorum +
held-out count) and the list of decisions I made.

The bug you'd ship

Say you're building an app that makes invoices. You add up the line items and show a total, like $1,204.50. Easy. You write total.toFixed(2) and move on.

But that one line is wrong in ways you won't notice until a customer does. Try it yourself:

# what you'd expect  →  what toFixed(2) actually gives
(2.675).toFixed(2)    "2.68"?  no → "2.67"
(10.995).toFixed(2)   "11.00"? no → "10.99"
(1.005).toFixed(2)    "1.01"?  no → "1.00"

Those are real cents, going missing, on real invoices. You didn't write that bug. It came from the way computers store numbers: 2.675 isn't really 2.675 inside the machine — it's a hair less, so it rounds down. Almost every money bug starts here.

How rederive looks at it

rederive doesn't treat "invoice app" as one big lump. It breaks it into small, named jobs — each one a thing the world already knows how to do:

  • round money — turn 2.675 into a clean 2-decimal value
  • format money — show it as $1,204.50 (or ¥1,205, or German 1.204,50 €)
  • format the date — show the invoice date
  • number the invoice — INV-0001

For each job, rederive finds the packages people normally reach for, and runs them all on the same inputs. Then it watches two things: where the packages agree, and where they disagree.

Where they agree, that's just how the job works — no decision to make. Where they disagree, that's a real fork in the road, and picking a branch changes your customer's total. rederive can't pick for you. So it stops and asks. That's the question.

The questions it asks you

On the invoice app, rounding money alone had two forks. Here's the first one, the way it reaches you:

When a value lands exactly halfway — like 2.345 — which way should it round?

half-up: always up → 2.35  ·  banker's: to the nearest even → 2.34

That doesn't look like a big deal until you're adding thousands of lines. "Always up" quietly tilts every total in one direction; "banker's" (round half to even) cancels out over many rows, which is why finance uses it. This is not a style choice. It's a question about who decides what "correct" means for your money — you, an accounting standard, or whichever package you happened to install.

The honest answer isn't "whatever, pick one." It's to name a source. On this app the ruling was: banker's rounding, and — the second fork — sum the exact numbers and round once at the end, never round each line and then add. Both were written down as decisions made on purpose:

The forkThe choiceWhy
Ties (2.345)banker's / half-evendoesn't drift totals; the finance standard
When to roundonce, at the totalround-per-line and round-once give different totals
Bad inputs (NaN, ∞)pass through, don't crashmatches the trusted reference's real behavior

The surprise

Now the part that made us sit up. The next job was format money — show the number with the right symbol, the right separators, the right number of decimals for the currency (US dollars have 2, yen has 0, Bahraini dinar has 3).

The popular packages for this — currency.js, numeral, accounting — were each about to add a dependency to your project. And they'd give you formatting that's US-centric and, in places, just wrong (yen with cents, for example).

The correct answer was already sitting in your language, free, called Intl. It knows every currency and locale, because it ships with the rules the whole industry agrees on. The entire verified job is one line:

// format_currency — the whole thing. Zero packages.
export default function formatCurrency(amount, code, locale = 'en-US') {
  return new Intl.NumberFormat(locale, { style: 'currency', currency: code }).format(amount);
}
This is the moment it clicks. The right answer was fewer dependencies, not more — and it was more correct at the same time. rederive didn't just remove a package. It showed us the package was never buying us anything. Deleting it was the fix.

How it proves the code

Removing packages only helps if the replacement is actually right. So here's the part that makes it safe to trust. For each job, rederive:

  1. writes down the correct answers by running a trusted reference — never by hand (for rounding, that reference was decimal.js in banker's mode);
  2. hides some of those answers as a secret held-out set;
  3. rebuilds the job from scratch three separate times, blind — each rebuild never sees the answer key or the other rebuilds;
  4. accepts the code only if all three agree, including on the hidden answers they were never shown.

Three blind rebuilds landing on the same behavior, on inputs none of them saw, is very hard to fake. That's the difference between correct code and a confident guess. Here's what the rounding job scored:

# money_round — final result from the real run
quorum        3 / 3 agree
checked on    18 shown + 19 held-out (never seen during the build)
anchored to   decimal.js, banker's rounding  # the chosen source of truth
ships         0 dependencies

And format_currency scored the same way: 3 / 3 agree, checked on 15 held-out cases, zero dependencies — one line of code doing what three packages did worse.

What you're left with

An invoice app that ships no dependencies at all, is provably right on the money math, and comes with a short list of every decision that was made on purpose — banker's rounding, round once, medium date format, and so on. If someone asks six months from now "why does it round this way?", the answer isn't "someone decided." It's written down, with the run that proves it.

That's the whole idea, small enough to hold in your hand. The chapters after this one show the other shapes it takes — getting a single package out of an existing project, catching a bug hiding in code you're about to migrate, clearing a whole dependency tree, and knowing when to stop and set something aside instead of pretending it's safe.

Chapter 2

Get one package out of your tree

In January 2022, the person who maintained colors — a tiny package inside tens of thousands of projects — pushed an update on purpose that printed garbage in an endless loop. Apps broke all over the world overnight. The people running them did nothing wrong. They just trusted a name.

That's the deal with every dependency. You're not really trusting the code you read once. You're trusting whoever is allowed to publish the next version, forever.

rederive gets you out of that deal, one package at a time. Take colors. It reads the last good version — 1.4.0, the release right before the sabotage — writes down exactly what it does, deletes it, and builds it again from scratch — blind, three times, checked on hidden inputs none of the rebuilds saw. (That written-down description of what the code does is its SIR.)

# colors → the "strip" function, from the real run
source        colors@1.4.0   # last release before the 2022 sabotage
quorum        3 / 3 agree
checked on    10 shown + 10 held-out
original      deleted — the rebuild never got to copy it
ships         0 dependencies

Now you have a colors that does exactly what the good version did, ships nothing underneath it, and cannot be changed by someone you've never met. You kept the behavior. You ship none of the package. Nobody can sabotage a copy you own and proved yourself.

Chapter 3

The bug you'd introduce

Sometimes the code you want to replace has a quirk — not a clean bug, but an odd behavior that other code quietly leans on. Rewrite it "the right way" and you break those callers, and you won't find out until a customer does.

Here's a real one. slugify turns "Hello World!" into "hello-world". It has an option called remove. The original decides whether to use your value like this:

// the original slugify — note the || (a "truthiness" check)
removePattern = options.remove || DEFAULT_PATTERN

// the "cleaner" rewrite everyone reaches for — and it's WRONG
removePattern = options.remove !== undefined ? options.remove : DEFAULT_PATTERN

They look the same. They are not. With the original, passing an empty string, or false, or 0 all fall back to the default. The "cleaner" version treats an empty string as a real value. Somebody's slugs quietly change, and nothing errors out to warn them.

rederive never rebuilds from what code should do. It rebuilds from what the code does do, proven by running the original. So it surfaces the real question: is this quirk the spec, or a bug to fix? On slugify, real callers depend on it — so it's kept, on purpose, and written down as a decision. A hand-port guesses at what the author meant. rederive copies what the code does and makes you choose, out loud, which quirks are load-bearing.

Chapter 4

The whole tree, and the finish line

One package is nice. Your real problem is the tree — a single npm install pulls in a hundred packages you will never read.

rederive can clear the whole tree, but it starts with a rule you cannot skip: run your tests first, and write down the number. Say your suite reports "182 passing" — write that down. That number is now the finish line. Nothing is done until it still passes.

Take del, a tool for deleting files — fittingly, the one you most want to be sure about. It leans on a small tree: slash, is-path-cwd, is-path-inside, is-glob, micromatch. rederive re-derives each one, leaves first, and drops verified copies into your project:

# del's dependency tree, re-derived — from the real run
slash          3.0.0 → 3/3 agree, checked on 400 held-out inputs
is-path-cwd    3.0.0 → 3/3 agree, checked on 500 held-out inputs
is-path-inside 4.0.0 → 3/3 agree
...            each one proven, then vendored with its provenance

Then it removes those packages from package.json, rewires the imports, and re-runs your tests. Same number in, same number out — or it isn't finished. The baseline isn't bureaucracy. It's the whole promise in one line: your app behaves exactly the same, minus the packages you were trusting on faith.

Chapter 5

No tests? Preflight first

Everything so far needs a source of truth to check against. But what if the code you want to move has no tests, or only tests the happy path? Then "it passed" means nothing — you had nothing real to pass.

rederive's answer is Preflight. Before it rebuilds anything, it builds a strong answer key from the real code — and, the part that matters, it measures how strong the answer key is. Most people never do that second part. That's the whole point of this chapter.

We'll follow one real run: cookie, a small, widely-used package that reads and writes browser cookie strings. Every number below is from that run.

Where do the questions come from?

If there are no tests, what do you even feed the code? Preflight gets its inputs from two places:

  • The rules the code is supposed to follow. Cookies have an official rulebook (it has a boring name, RFC 6265). Preflight walks that rulebook and makes an input for every case it names — the normal ones and the picky edge ones.
  • Pure garbage. On top of the sensible inputs, it throws junk at the code on purpose: emoji, a single dot, a 400-character wall of nonsense, random broken text. Real users send weird things, so the answer key should know what the code does with weird things too.

For each input, it writes down what the real code does — never a hand-guessed answer. That list of input-and-answer pairs is the answer key.

Does it hold up under garbage?

The junk inputs aren't just for show. They check a promise that matters: the code should never crash, on any string, ever, and should always be able to read back what it wrote. On cookie, all six of those promises held — no input, however broken, could make it throw or lose data.

How strong is the answer key, really?

Now the honest part. A green test run feels good but proves little — the test might run a line without ever checking the result. So Preflight breaks the code on purpose and watches whether the answer key notices. This is called mutation testing: it makes hundreds of slightly-wrong copies of the code — flip a < to <=, swap a + for a - — and re-runs the answer key against each one. Every wrong copy the key catches is a point in its favor. Every wrong copy that slips through unnoticed is a hole, in that exact spot.

# cookie — the mutation run, real numbers
wrong copies made        361
caught by the answer key 332
slipped through          13
score                    96.4%   # the pass mark is 90%

Wait — what about the 13 that slipped through?

This is the question everyone asks, and the answer is the best part. You don't shrug at the 13. You look at every single one. On cookie, each of the 13 turned out to be a wrong copy that can't actually change the answer — like deleting a shortcut when the long way around gives the exact same result. They're changes with no effect. Each one was checked by hand and written down with the reason.

Here's the real finish line. It isn't "96%." It's zero unexplained holes. Every wrong copy is either caught, or shown by hand to be a change that cannot matter. Nothing is left as a shrug. On cookie: 0 unexplained

And if the score is low?

Sometimes a slipped-through copy is a real hole — a genuine change the answer key missed. That's not a failure, it's a finding, and it has a fix: find the one input that makes the wrong copy behave differently, and add it to the key. On cookie, the first pass scored 95.8% with 15 slipping through; adding two pointed inputs caught two real ones and lifted it to 96.4%. You either catch it or prove it can't matter — there's no third option where you just hope.

Isn't this just code coverage?

No, and this is the trap Preflight is built to escape. Code coverage only tells you a line ran during testing. It says nothing about whether anyone checked the result. cookie already had 100% coverage — every line, every branch, run — and mutation testing still found real holes. Coverage says "the line ran." Mutation testing says "if that line were wrong, the answer key would catch it." Only the second one is worth trusting.

So Preflight turns "I have a bad feeling about these tests" into a number and a short list of checked-off holes. If the number is too low to trust, that's a finding — and you fix it before you build anything on top of it, instead of finding out later from a customer.

Chapter 6

When to quarantine

The most useful thing rederive can say is no.

Some code can't be soundly verified — not because rederive is weak, but because the thing genuinely has no fixed answer to check. Anything driven by randomness, by real-world side effects, or by timing that a function you pass in controls.

Take p-map: it runs many async jobs with a limit on how many run at once. Its behavior depends on a function you hand it and on timing. There is no clean "this input gives that output" to freeze. The correct move is to quarantine it — set it aside clearly, and tell you why — instead of pretending.

In one of our own runs, an agent didn't do that. It tried to hand-build a checker for p-map, got the concurrency wrong, and left this comment sitting in its own code:

# wrong: naive Promise.all —

That's the failure a quarantine prevents. A quarantine is not rederive giving up. It's rederive telling you exactly where the proven ground ends — so you never mistake a confident guess for a proof. Knowing the edge of what you've verified is the whole point of it.
